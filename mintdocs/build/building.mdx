This page covers everything you run regularly while developing — keeping dependencies in sync after a pull, regenerating Dart code after model changes, running the app locally, and producing web and Android builds. For first-time machine setup, see [First Time Setup](/build/first-time-setup).

## PHBServer — sync after pulling

Run this every time you pull changes or switch branches:

```bash
./dev-sync.sh
```

It performs the following steps:

```bash
phbcli stop          # release Windows file lock on the binary before overwriting
cd phbserver
uv sync
uv tool install --editable phbcli --upgrade --force
```

<Warning>
`uv sync` alone is **not enough** after code changes. `phbcli` is a `uv tool` — its entry-point scripts are baked at install time. If `pyproject.toml` or the package structure changes, the old script will point at a stale or missing module and fail with `ModuleNotFoundError`. Always run `./dev-sync.sh` after pulling. On Windows the server must be stopped first so the binary file lock is released.
</Warning>

## Flutter — code generation

Several packages (`freezed`, `json_serializable`, `riverpod_generator`, `drift`) require a code generation step. Run this from `device_apps/` any time you add or modify a model, provider, or database table:

```bash
fvm dart run build_runner build --delete-conflicting-outputs
```

During active development, use watch mode so code is regenerated automatically on file save:

```bash
fvm dart run build_runner watch --delete-conflicting-outputs
```

<Note>
Always run `build_runner build` once after `fvm flutter pub get` on a fresh clone, before attempting any build or run command.
</Note>

## Flutter — run for development

```bash
cd device_apps
fvm flutter run
```

Add `-d <device-id>` to target a specific device or emulator. Use `fvm flutter devices` to list available targets.

## Flutter — build for web

From `device_apps/`:

```bash
fvm flutter build web
```

Output is written to `build/web/`. To host under a sub-path (e.g. `/app/`):

```bash
fvm flutter build web --base-href /app/
```

<Note>
You may see Wasm dry-run warnings about `flutter_secure_storage_web` using `dart:html`. These are informational — the standard JavaScript/CanvasKit build succeeds normally. Wasm support for that package is a known upstream limitation.
</Note>

## Flutter — build for Android

From `device_apps/`:

```bash
# Debug APK — quickest for local testing
fvm flutter build apk

# Split APKs per ABI — smaller files, suitable for direct device install
fvm flutter build apk --split-per-abi

# Release App Bundle — required for Google Play upload
fvm flutter build appbundle --release
```

Output APKs are written to `build/app/outputs/flutter-apk/`.

<Note>
`android/gradle.properties` sets `kotlin.incremental=false`. This is required when the Flutter Pub cache (`C:\Users\...\AppData\Local\Pub\Cache`) and the project (`D:\...`) are on different Windows drives — the Kotlin incremental compiler cannot compute relative paths across drive letters and fails the build otherwise.
</Note>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Build fails: 'Error when reading *.freezed.dart' or '*.g.dart'">
    The generated source files do not exist yet. Run code generation first, then retry the build:

    ```bash
    fvm dart run build_runner build --delete-conflicting-outputs
    ```

    This must be run at least once after a fresh clone and any time you add a new `@freezed` model or `@riverpod` provider.
  </Accordion>

  <Accordion title="Android build fails: 'this and base files have different roots' (Kotlin cross-drive error)">
    This happens on Windows when the Flutter Pub cache is on a different drive than the project (e.g. cache on `C:`, project on `D:`). The Kotlin incremental compiler cannot resolve cross-drive relative paths.

    The fix is already applied in this repo (`android/gradle.properties` contains `kotlin.incremental=false`). If you see this error, verify that line is present:

    ```properties
    kotlin.incremental=false
    ```

    If you have removed it or are starting from a different branch, add it back and run `fvm flutter clean` before retrying.
  </Accordion>

  <Accordion title="Android build fails with 'Daemon compilation failed: null' and a long Kotlin stack trace">
    This is a symptom of the same cross-drive Kotlin incremental cache issue described above. Check that `kotlin.incremental=false` is set in `android/gradle.properties`, then clean and rebuild:

    ```bash
    fvm flutter clean
    fvm flutter build apk
    ```
  </Accordion>

  <Accordion title="Web build shows Wasm warnings about flutter_secure_storage_web">
    These warnings are printed by Flutter's Wasm dry-run check and are expected. The standard JavaScript/CanvasKit web build completes successfully regardless. You can suppress the warnings with `--no-wasm-dry-run`, but they do not indicate a real build problem.
  </Accordion>

  <Accordion title="ModuleNotFoundError when running phbcli after pulling">
    The server code changed but the installed tool entry-point still points to the old module layout. Run the sync script to reinstall:

    ```bash
    ./dev-sync.sh
    ```

    On Windows, stop the server first (`phbcli stop`) so the binary file lock is released before the reinstall.
  </Accordion>

  <Accordion title="fvm flutter run picks up stale generated code after adding a new model">
    Hot reload does not trigger code generation. After adding a new `@freezed` model or `@riverpod` provider, stop the app, run `build_runner build`, then restart:

    ```bash
    fvm dart run build_runner build --delete-conflicting-outputs
    fvm flutter run
    ```
  </Accordion>
</AccordionGroup>
