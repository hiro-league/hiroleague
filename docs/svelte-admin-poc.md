# HiroAdmin SvelteKit POC

## Purpose

This document tracks the Svelte-based HiroAdmin migration. It started as a POC and now contains the first real migrated page.

The current NiceGUI admin remains unchanged. The SvelteKit app lives beside the Python packages, builds into a small static asset package, and can later be mounted by FastAPI when a migrated admin route is ready.

The Svelte admin includes real backend integrations. The dashboard calls `GET /admin-next/api/workspaces`, and the Server page exposes Workspaces and Gateways tabs backed by the existing NiceGUI admin services.

## Goals

- Keep the existing NiceGUI admin untouched.
- Put frontend source outside the Python package in `admin_frontend/`.
- Package only compiled static assets inside `hirocli.admin_svelte`.
- Make local frontend iteration fast with Vite dev server.
- Establish a scalable SvelteKit folder structure before migrating real pages.
- Provide a page-by-page migration path from NiceGUI controllers/components to Svelte components and Python API routes.
- Migrate real admin pages incrementally while preserving NiceGUI for comparison.

## Non-Goals

- Replacing NiceGUI in this POC.
- Migrating the log viewer yet.
- Adding admin authentication changes.
- Adding a production API contract for every admin feature.

## Folder Layout

```text
admin_frontend/
  src/
    routes/
      +layout.svelte
      +layout.ts

      server/

    lib/

      features/
        server/

          GatewaysTab.svelte
      shell/
        AdminShell.svelte
        nav.ts
      ui/
        Modal.svelte
    styles/
      app.css

hiroserver/hirocli/src/hirocli/admin_svelte/
  __init__.py
  api.py
  static/
    # generated by npm run package:python
  static_server.py
```

`admin_frontend/` is the editable SvelteKit source tree. It is not imported by Python.

`hirocli.admin_svelte.static` is the packaged output location. It is the only frontend part intended for the Python wheel.

## Local Development

Install frontend dependencies:

```bash
npm --prefix admin_frontend install
```

Run the SvelteKit dev server:

```bash
npm --prefix admin_frontend run dev
```

Open:

```text
http://127.0.0.1:5173/admin-next/
```

This does not require rebuilding or restarting Hiro for frontend-only edits.

During development, Vite proxies Svelte API calls to a running Hiro admin server:

```text
http://127.0.0.1:5173/admin-next/api/workspaces
  -> http://127.0.0.1:18083/admin-next/api/workspaces
```

The default proxy target is `http://127.0.0.1:18083`, which is the slot-0 admin port. For another admin port, set `HIRO_ADMIN_ORIGIN` before starting Vite:

```powershell
$env:HIRO_ADMIN_ORIGIN = "http://127.0.0.1:18087"
npm --prefix admin_frontend run dev
```

With that setup, the normal workflow is:

1. Start Hiro once with `hiro start --admin`.
2. Start Vite once with `npm --prefix admin_frontend run dev`.
3. Edit Svelte files and let Vite hot reload the browser.
4. Rebuild with `npm --prefix admin_frontend run package:python` only when you want Hiro itself to serve the latest static assets.

## Packaging Flow

Build static assets into the Python package:

```bash
npm --prefix admin_frontend run package:python
```

The SvelteKit static adapter writes to:

```text
hiroserver/hirocli/src/hirocli/admin_svelte/static/
```

The Python package config includes those generated files through Hatch:

```toml
[tool.hatch.build.targets.wheel]
packages = ["src/hirocli"]
artifacts = ["src/hirocli/admin_svelte/static/**"]
```

Release automation should run the frontend package step before building the Python wheel.

```

```

```text

```

The NiceGUI admin remains mounted at `/`. The Svelte POC is isolated under `/admin-next/`.

## FastAPI Mount

The helper in `hirocli.admin_svelte.static_server` mounts the built static app:

```python
from fastapi import FastAPI
from hirocli.admin_svelte.static_server import mount_admin_svelte_static

app = FastAPI()
mount_admin_svelte_static(app)
```

The existing NiceGUI admin bootstrap mounts this helper under `/admin-next/` only when built assets are present. If assets are missing, NiceGUI still starts normally and logs a warning.

The API routes are registered from `hirocli.admin_svelte.api` and call existing admin services.

## Migrated Server Page

The first real migrated page is:

```text
/admin-next/server/
```

It keeps NiceGUI untouched and mirrors existing admin behavior through API calls to:

- `WorkspaceService`
- `GatewayService`

### Workspaces Tab

Implemented:

- list workspaces
- create workspace
- edit workspace name, gateway URL, and default flag
- remove workspace with optional purge
- start workspace
- stop workspace
- restart workspace with optional admin UI
- run workspace setup
- show setup-generated public key
- view public key
- copy public key
- regenerate public key
- open workspace folder

### Gateways Tab

Implemented:

- list gateway instances
- create gateway instance
- start gateway
- stop gateway
- remove gateway with optional purge
- advanced create options for host, default, autostart, and elevated task

## Migration Model

Current NiceGUI shape:

```text
NiceGUI page/controller/components
  -> admin service
    -> tools/domain
```

Target Svelte shape:

```text
Svelte route/components/stores
  -> /admin/api/* FastAPI routes
    -> existing admin service
      -> tools/domain
```

The existing service layer is already mostly UI-free, so most business behavior should survive the migration. NiceGUI controllers and components become the disposable layer.

## Recommended Feature Structure

As pages migrate, use a feature-sliced Svelte layout:

```text
admin_frontend/src/lib/
  api/
    client.ts
    logs.ts
    characters.ts
  features/
    logs/
      LogTable.svelte
      LogToolbar.svelte
      LogDetailPanel.svelte
      log-store.svelte.ts
      log-types.ts
    characters/
      CharacterBrowse.svelte
      CharacterEditor.svelte
      character-store.svelte.ts
      character-types.ts
  shell/
    AdminShell.svelte
    Sidebar.svelte
    Header.svelte
  ui/
    Button.svelte
    Dialog.svelte
    Select.svelte
    StatusBadge.svelte
```

Rules:

- Routes stay thin.
- Feature state lives in feature stores.
- API calls live under `lib/api`.
- Reusable widgets live under `lib/ui`.
- Python remains the owner of business logic, workspace access, filesystem access, and tool execution.

## First Real Migration Candidate

The log viewer is the best proof page because it exercises the hard parts:

- incremental tailing
- pause/resume
- filtering
- sorting
- custom cell rendering
- row selection
- detail panel rendering
- user preference persistence

Suggested API surface:

```text
GET  /admin/api/logs/layout
POST /admin/api/logs/tail
POST /admin/api/logs/search
```

The browser should own visible rows, selected row, active filters, sort order, auto-scroll, and panel state. Python should own log reading, offsets, search, workspace resolution, and normalized row payloads.
